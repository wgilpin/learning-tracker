# Data Model: Multi-User Isolation (004)

## New Tables

### `users`

Represents a registered account. Created only via invitation code redemption.

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `id` | UUID | PK, default gen_random_uuid() | |
| `email` | VARCHAR(320) | NOT NULL, UNIQUE | Normalised to lowercase |
| `password_hash` | VARCHAR(60) | NOT NULL | bcrypt output (always 60 chars) |
| `is_active` | BOOLEAN | NOT NULL, DEFAULT true | False = deactivated; blocks login |
| `created_at` | TIMESTAMPTZ | NOT NULL, DEFAULT now() | |
| `deactivated_at` | TIMESTAMPTZ | NULL | Set when operator calls `manage deactivate-user` |

**Indexes**: `users_email_idx` UNIQUE on `email`.

---

### `invitation_codes`

Single-use tokens generated by the CLI. Consumed at account creation.

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `code` | VARCHAR(64) | PK | URL-safe random token (e.g. 32-byte hex) |
| `is_used` | BOOLEAN | NOT NULL, DEFAULT false | Set to true when consumed |
| `created_at` | TIMESTAMPTZ | NOT NULL, DEFAULT now() | |
| `used_at` | TIMESTAMPTZ | NULL | Timestamp of redemption |
| `used_by_user_id` | UUID | FK → users.id, NULL | Populated on redemption |

**Invariant**: Once `is_used = true`, the row is immutable. No update path other than the redemption transaction.

---

### `user_source_refs`

Per-user, per-topic link to a shared `Source` record. Replaces the direct `Source.topic_id` FK as the ownership mechanism.

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `id` | UUID | PK | |
| `user_id` | UUID | NOT NULL, FK → users.id ON DELETE CASCADE | |
| `source_id` | UUID | NOT NULL, FK → sources.id ON DELETE CASCADE | Physical file record |
| `topic_id` | UUID | NOT NULL, FK → topics.id ON DELETE CASCADE | The user's topic this source is filed under |
| `created_at` | TIMESTAMPTZ | NOT NULL, DEFAULT now() | |

**Indexes**:
- UNIQUE on `(user_id, source_id, topic_id)` — prevents duplicate refs.
- Index on `source_id` — for efficient ref-count queries (`COUNT(*) WHERE source_id = ?`).
- Index on `(user_id, topic_id)` — for listing sources by topic per user.

**Reference counting**: `SELECT COUNT(*) FROM user_source_refs WHERE source_id = ?`. When count reaches 0 after a delete, the `Source` row and its ChromaDB collection are removed.

---

## Modified Tables

### `topics` — add `user_id`

| Column | Change |
|--------|--------|
| `user_id` | NEW: UUID, NOT NULL, FK → users.id ON DELETE CASCADE |

All queries on `topics` must filter `WHERE user_id = :current_user_id`. The existing cascade (topics → syllabus_items → atomic_chapters → margin_comments) means all child data is implicitly owned by the user via the topic.

**Index**: `topics_user_id_idx` on `user_id`.

---

### `sources` — restructure ownership

| Column | Change |
|--------|--------|
| `topic_id` | REMOVED — ownership moved to `user_source_refs.topic_id` |
| `content_hash` | Unique constraint changed from `(topic_id, content_hash)` to global UNIQUE on `content_hash` alone |

All other columns (`source_type`, `index_status`, `title`, `content`, etc.) are unchanged — they describe the physical document, not the ownership.

**Effect**: `Source` becomes a global shared record. Multiple users see the same `Source` row but never know it. Each has their own `UserSourceRef` pointing to it.

---

## Unchanged Tables

`syllabus_items`, `atomic_chapters`, `margin_comments`, `chapter_sources` — ownership is derived transitively through `topics.user_id`. No direct `user_id` column needed on these tables; queries always start from `topics WHERE user_id = ?` and join down.

---

## Entity Relationships

```
users (1) ──── (N) topics
users (1) ──── (N) invitation_codes [used_by_user_id, nullable]
users (1) ──── (N) user_source_refs
topics (1) ──── (N) syllabus_items
topics (1) ──── (N) atomic_chapters
topics (1) ──── (N) user_source_refs [topic_id]
syllabus_items (1) ──── (0|1) atomic_chapters
atomic_chapters (1) ──── (N) margin_comments
atomic_chapters (N) ──── (N) sources [via chapter_sources]
user_source_refs (N) ──── (1) sources
sources (1) ──── (1) chromadb collection [source_{id.hex}]
```

## ChromaDB Topology Change

| Before | After |
|--------|-------|
| One collection per topic: `topic_{topic_id.hex}` | One collection per source: `source_{source_id.hex}` |
| Chunk metadata: `{source_id, topic_id, chunk_index}` | Chunk metadata: `{source_id, chunk_index}` |

**Query path**: To search across a user's topic sources: (1) load `UserSourceRef` rows for `(user_id, topic_id)` → get list of `source_id`s; (2) query each `source_{id}` collection; (3) merge results. The `chroma.py` service is updated to use per-source collections.

**Migration**: Existing `topic_*` collections are orphaned after migration 0008. A one-time cleanup script should drop them. The seed user's sources will need to be re-indexed (index_status reset to PENDING) since collection names change.
